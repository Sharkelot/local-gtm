# Security model

## Overview

Local GTM is designed as a **private production deployment** sourced from a
**public portfolio repository**. Security controls enforce this separation at
every layer: source control, CI/CD, container registry, deployment, network, and
application authentication.

## Source control controls

- `.env` and `.env.*` files are gitignored except `.env.example`.
- All example files use placeholder values (`crm.example.com`, `DATABASE_SERVER`, etc.).
- Real internal addresses (RFC 1918), passwords, tokens, and certificates appear
  only in local files outside the repository.
- `SECURITY.md` and `CONTRIBUTING.md` document the public-repository rules.

## CI/CD security

- CI runs exclusively on **GitHub-hosted Ubuntu runners** with `contents: read`
  for pull requests. No self-hosted runners are registered for this repository.
- External Actions are pinned to full commit SHAs with release-tag comments.
- Pull-request workflows have no access to repository secrets.
- The publish workflow triggers only after CI succeeds on protected `main` and
  uses the ephemeral `GITHUB_TOKEN` with `packages: write` scope.
- Secret scanning (Gitleaks), filesystem vulnerability scanning (Trivy), and
  dependency auditing run on every CI invocation.

## Image security

- Container images are built and published to GHCR only after CI passes on `main`.
- Images are tagged with the immutable Git commit SHA. Mutable `main` tags exist
  for discovery but production must deploy by digest.
- Images do not embed runtime environment files, private addresses, or
  credentials. All configuration is injected at deploy time.
- Dockerfiles use distroless-inspired patterns (`bookworm-slim`), drop root,
  enable `read_only: true` and `no-new-privileges`, and mount `tmpfs` for
  ephemeral writes.

## Deployment security

- A dedicated `local-gtm-deployer` CT polls GHCR outbound. No inbound webhook
  or self-hosted runner connects to the Proxmox environment.
- The deployer authenticates to application CTs via dedicated SSH keys with
  `BatchMode=yes`, `IdentitiesOnly=yes`, and known-host pinning. It stores no
  GitHub write credential.
- The deployer never executes pull-request code. It pulls only operator-approved
  digests from GHCR.
- The deployer validates application health after deployment and rolls back to
  the previous digest on failure.

## Network security

- Cloudflare Tunnel is the only public ingress path. No direct WAN port
  forwarding reaches production CTs.
- Caddy terminates TLS and reverse-proxies to the internal web service.
- Database (PostgreSQL), Redis, document storage (MinIO), secrets (OpenBao),
  malware scanning (ClamAV), and Keycloak admin ports remain LAN-only.
- Internal Compose networks isolate application, data, and observability planes.

## Application authentication

- All CRM access requires Keycloak authentication. Anonymous access is not
  permitted in production configurations.
- Keycloak self-registration is disabled (`registrationAllowed=false`). Only
  pre-created, administrator-approved users may authenticate.
- The Keycloak client is confidential (non-public), uses PKCE with S256, and
  restricts redirect URIs to the exact application callback.
- Web origins are restricted to the application domain.
- Session tokens use `__Secure-` prefix, `httpOnly`, `sameSite=lax`, and TLS-only
  transmission in production.
- Admin pages require explicit realm roles assigned by a Keycloak administrator.

## Identity provider

- Keycloak runs as a dedicated container with its own PostgreSQL schema.
- Brute-force protection is enabled (5 failures triggers backoff).
- Password policy requires minimum 14 characters with digit, lowercase,
  uppercase, and special character.
- TOTP (time-based one-time password) MFA is required for all users
  (`CONFIGURE_TOTP` required action).
- The realm admin hostname is separate from the user-facing auth hostname and
  restricted by source IP (CIDR allowlist).

## Data protection

- PostgreSQL connections use SCRAM-SHA-256 password encryption.
- In-production TLS between CTs uses a private CA (generated by
  `deploy/split/generate-private-pki.sh`).
- Every tenant-owned record includes `tenant_id`. RLS (row-level security)
  enforces tenant isolation at the database level.
- Financial ledger entries are immutable; corrections use
  auditor-linked reversals.
- Uploaded documents are quarantined until ClamAV malware scanning completes.
- OpenBao (Vault fork) manages secrets with workload-specific short-lived tokens.

## Health endpoints

- `/api/health/live` — process is running (no dependency check).
- `/api/health/ready` — required dependencies (database) are reachable.
- `/api/health/smoke` — authenticated dependencies and environment
  configuration are correct.
- Health responses do not expose secret values or internal infrastructure
  details.

## Compliance alignment

Security controls map to SOC 2 trust service criteria:

| Control                      | SOC 2 criterion               |
| ---------------------------- | ----------------------------- |
| Tenant RLS + audit trail     | CC6.1 (logical access)        |
| Immutable ledger + reversals | CC3.4 (record integrity)      |
| Malware quarantine           | CC6.7 (malware protection)    |
| MFA + strong passwords       | CC6.1 (authentication)        |
| Pull-based deployment        | CC7.1 (change management)     |
| Read-only CI permissions     | CC6.1 (segregation of duties) |
| OpenBao secrets management   | CC6.2 (secrets management)    |

See [docs/compliance/soc2-control-and-evidence-outline.md](compliance/soc2-control-and-evidence-outline.md).
